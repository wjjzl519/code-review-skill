# Code Review 技能

自动化代码审查工具，基于阿里巴巴Java开发手册和业界最佳实践，提供全面的代码质量检查。

## 核心检查维度

### 1. 边界条件检查
- **数组/集合边界**：检查数组越界、空集合、单元素集合等边界情况
- **数值边界**：检查整数溢出、除零、浮点精度问题
- **字符串边界**：空字符串、null、超长字符串
- **时间边界**：时区问题、闰年、月末日期
- **并发边界**：竞态条件、死锁、资源竞争

### 2. 异常处理
- **异常捕获规范**：
  - 不要捕获 Exception 和 Throwable，应捕获具体异常
  - 避免空 catch 块
  - finally 块中不要使用 return
  - 不要在 finally 中抛出异常
- **异常传播**：检查是否正确向上层传播异常
- **资源释放**：检查是否使用 try-with-resources 或手动关闭资源
- **日志记录**：异常应记录完整堆栈信息
- **try 内 early return**：`try` 块内的 `return ResultData.error(...)` 不会进入 `catch`；若 `try` 外或之前有副作用（DB 写入、加锁等），须单独审查该分支是否对称回滚，不可因存在 `catch` 就认定「失败已清理」

### 3. 安全漏洞检查
- **SQL注入**：检查是否使用参数化查询
- **XSS攻击**：检查输出是否进行转义
- **敏感信息泄露**：检查日志中是否包含密码、token等
- **权限校验**：检查是否缺少权限验证
- **加密算法**：禁止使用MD5、SHA1等弱加密算法
- **随机数安全**：使用 SecureRandom 而非 Random

### 4. 性能问题
- **循环优化**：
  - 避免在循环中进行数据库操作
  - 避免在循环中创建大量对象
  - 循环内避免正则表达式编译
- **集合使用**：
  - 指定集合初始容量避免扩容
  - 使用 StringBuilder 而非字符串拼接
- **资源泄漏**：检查连接、流、锁是否正确释放
- **N+1查询**：检查是否存在批量查询优化机会

### 5. 代码复杂度
- **圈复杂度**：单个方法圈复杂度不应超过 10
  - 计算方法：决策点数量 + 1（if/else/for/while/case 等）
  - 超过10时建议拆分方法
- **方法长度**：单个方法不超过 80 行
- **类长度**：单个类不超过 500 行
- **参数数量**：方法参数不超过 5 个
- **嵌套深度**：代码嵌套不超过 4 层

### 6. 命名规范（阿里巴巴规范）
- **类名**：UpperCamelCase，DO/BO/DTO/VO/DAO/Service/Controller 等后缀
- **方法名**：lowerCamelCase，动词开头（get/set/is/has/save/update/delete）
- **常量**：全大写，下划线分隔
- **变量**：lowerCamelCase，见名知意
- **包名**：全小写，点分隔
- **禁止拼音命名**：除特定领域词汇（如 taobao、hangzhou）

### 7. 副作用与出口对称性（有副作用时必须做）

凡方法内存在 **不可逆副作用**（DB insert/update、分布式锁、文件落盘、MQ 发送、外部 API 调用等），必须做出口分析，禁止仅凭「有 try-catch」或「catch 里有 cleanup」下结论。

**分析步骤**（写在审查过程中，关键方法必须执行）：

1. **标出副作用点**：列出所有产生持久状态变更的操作及行号
2. **枚举全部出口**：逐个找出 `return`、`throw`、`catch` 中的返回/抛出；区分：
   - 异常退出（进入 `catch`）
   - 正常错误返回（`try` 内 `return error`，不抛异常）
   - 成功退出
3. **逐出口核对**：对每个失败出口回答——是否需要回滚/补偿？当前代码是否已处理？
4. **区分覆盖范围**：结论必须三选一，禁止模糊表述：
   - **完全未处理**：所有失败路径均未清理
   - **部分未处理**：已处理的路径 + **未处理的具体行号/分支**（必须列出）
   - **已对称处理**：所有失败路径均已清理或业务上应保留

**报告表述约束**（强制）：

- ✅ 正确：「`catch` 中已 `removeById`（426 行），但 `try` 内 392、397 行 `return` 早退未删批次」
- ❌ 错误：「失败时未清理批次」（忽略了已存在的 `catch` 逻辑）
- 指出问题时，**先承认已有正确逻辑，再指出遗漏分支**
- 每个 🔴/🟠 问题须附 **证据行号**；无行号不得升级为严重问题

**严重性校准**：

- 仅影响低概率分支（如 `inputStream == null`）的局部遗漏 → 通常 🟠，不应默认 🔴
- 升级为 🔴 须同时满足：可复现路径明确 + 后果为数据损坏/安全/资损，并在报告中说明理由

## 审查流程

### 步骤1：代码接收
识别用户提供的代码，可能的形式：
- 直接粘贴的代码片段
- 上传的代码文件
- 指向代码文件的路径

### 步骤2：全面扫描
按照以下顺序进行检查：
1. 快速浏览整体结构
2. **副作用出口分析**（存在 DB 写入/加锁/文件/MQ 等方法时 **必做**，见「核心检查维度 §7」）
3. 边界条件检查
4. 异常处理检查（含 `try` 内 early return 与 `catch` 覆盖范围对比）
5. 安全漏洞检查
6. 性能问题检查
7. 复杂度计算
8. 命名规范检查
9. 对照《Java 开发手册（黄山版）》`alibaba-manual/` 中的【强制】与【推荐】条目（见下方加载策略）

### 步骤2.5：事实核验（出报告前必做）

在写入报告前，对每个待报问题自检：

1. **可反驳性**：若开发者指出「我在 X 行已经处理了」，我是否仍能成立？若不能，改写或降级
2. **路径完整性**：是否把「部分遗漏」误写成「完全未做」？
3. **严重性匹配**：严重级别是否与触发概率、影响面一致？
4. **用户质疑响应**：若被纠正，应明确承认原表述不精确，并给出修正后的分支级结论

### 步骤3：问题分级
将发现的问题分为四个级别：
- **🔴 严重（Critical）**：安全漏洞、资源泄漏、数据丢失风险
- **🟠 重要（Major）**：性能问题、异常处理不当、复杂度过高
- **🟡 一般（Minor）**：命名不规范、代码风格问题
- **🔵 建议（Suggestion）**：可优化点、最佳实践建议

### 步骤4：生成报告
使用结构化的 Markdown 格式输出审查报告。

## 规约加载策略

1. **第一轮（必做）**：基于上文「核心检查维度」独立扫描，不加载 reference
2. **第二轮（按需）**：根据上表读取 **1～3 个** 相关 reference 文件
3. **第三轮（深度）**：发现具体疑点时，在 reference 中定位对应条目，报告须引用：`《Java开发手册（黄山版）》· [章节] · [子节] · 第 N 条【强制/推荐】`
4. **优先级**：先查全部【强制】，再查【推荐】
5. **禁止**：一次性 Read 全部 `alibaba-manual/*.md`

手册更新后，运行 `scripts/regenerate-alibaba-manual.py` 重新生成 reference（需 `pip3 install pymupdf`）。

## 注意事项

1. **客观公正**：基于规范和最佳实践，不带主观偏见
2. **提供依据**：每个问题都要说明违反了哪条规范（优先引用 `alibaba-manual/` 中的具体条目编号）
3. **建设性**：不仅指出问题，更要给出解决方案
4. **鼓励为主**：认可好的代码实践，鼓励持续改进
5. **灵活性**：规范是指导而非教条，特殊场景可以说明
6. **完整性**：即使用户只提到"看看代码"，也要进行全面审查
7. **禁止模式匹配式审查**：不得用「先 save 再 try-catch」等惯用模式直接推断缺陷，必须按出口逐条验证
8. **精确表述优先**：宁可少报一条，不可把「部分遗漏」夸大为「完全缺失」；被用户纠正时，优先修正结论而非辩护原报告

## 反模式示例

**场景**：方法在 `try` 前 `save()`，`catch` 中 `removeById()`，`try` 内有两处 `return ResultData.error(...)`

| 审查结论 | 评价 |
|---------|------|
| 「失败时未清理，批次会残留」 | ❌ 忽略 `catch` 已清理异常路径 |
| 「异常路径已在 catch 清理，但 392、397 行早退未删批次」 | ✅ 分支级、可验证 |
| 标为 🔴 严重 | ⚠️ 通常过重，除非该早退分支高频且阻断核心业务 |

## 技能边界

### 适用场景
- PR 代码审查
- 提交前自审
- 代码质量诊断
- 重构前评估

### 不适用场景
- 业务逻辑正确性判断（需要业务上下文）
- 架构设计评审（超出单个代码片段范围）
- 代码生成或重写（应使用其他技能）
